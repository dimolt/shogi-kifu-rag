# テスト駆動開発（TDD）

## 哲学

**基本原則**: テストは実装の詳細ではなく、公開インターフェースを通じた振る舞いを検証すべきである。コードは全面的に変更されてもよいが、テストは変更されるべきではない。

**良いテスト** は統合テスト寄りであり、公開 API を通して実際のコードパスを実行する。テストが記述するのはシステムが「何をするか」であって、「どのように実装されているか」ではない。

良いテストは仕様書のように読める。

例:
「ユーザーは有効なカートでチェックアウトできる」

このテストを見れば、どの機能が存在するかが明確に分かる。

こうしたテストは内部構造に依存しないため、リファクタリング後も生き残る。

**悪いテスト** は実装に密結合している。

- 内部の協調オブジェクトをモックする
- private メソッドをテストする
- 外部手段で検証する
  - 例: インターフェース経由で確認せず、DB を直接参照する

危険信号は次のような場合である。

- 振る舞いは変わっていないのにリファクタリングでテストが壊れる
- 内部関数名を変更しただけでテストが失敗する

そのようなテストは振る舞いではなく実装をテストしている。

例については `tests.md` を参照し、モックの指針については `mocking.md` を参照すること。

## アンチパターン: 水平スライス

**すべてのテストを書いてから、すべての実装を書くことをしてはいけない。**

これは「水平スライス」と呼ばれる。

RED を「全テストを書く段階」、
GREEN を「全コードを書く段階」
と考えてしまうやり方である。

この方法は **質の低いテスト** を生み出す。

- 想像上の振る舞いをテストしてしまい、実際の振る舞いを検証しない
- ユーザー向けの振る舞いではなく、データ構造や関数シグネチャなど「形」をテストしてしまう
- テストが実際の変更に鈍感になる
  - 振る舞いが壊れても通る
  - 振る舞いが正常でも失敗する
- 実装を理解する前にテスト構造を固定してしまう

### 正しい方法

Tracer Bullet（試射弾）の考え方による **垂直スライス** を行う。

1 テスト → 1 実装 → 繰り返し

各テストは前サイクルで得た知見を反映して作成する。

直前にコードを書いたばかりなので、

- どの振る舞いが重要か
- どう検証すべきか

を正確に理解している。

```text
間違い（水平スライス）

RED:
  test1, test2, test3, test4, test5

GREEN:
  impl1, impl2, impl3, impl4, impl5


正しい方法（垂直スライス）

RED→GREEN:
  test1 → impl1

RED→GREEN:
  test2 → impl2

RED→GREEN:
  test3 → impl3

...
````

## ワークフロー

### 1. 計画

コードを書く前に行うこと:

* [ ] ユーザーと必要なインターフェース変更を確認する
* [ ] テストすべき振る舞いを確認し、優先順位を付ける
* [ ] Deep Module 化の機会を探す

  * 小さなインターフェース
  * 深い実装
* [ ] テストしやすいインターフェースを設計する
* [ ] テスト対象の振る舞いを列挙する

  * 実装手順ではない
* [ ] 計画についてユーザーの承認を得る

コードベースを調査する際は、

* プロジェクトのドメイングロッサリーを利用する
* テスト名やインターフェース用語をプロジェクト用語に合わせる
* 変更箇所周辺の ADR（Architecture Decision Record）を尊重する

まず次の質問を行う。

> 公開インターフェースはどのような形にすべきですか？
> どの振る舞いを最優先でテストしたいですか？

### 重要

**すべてをテストすることはできない。**

ユーザーと確認し、

* 本当に重要な振る舞い
* クリティカルパス
* 複雑なロジック

に集中すること。

あらゆる境界ケースを網羅しようとしてはいけない。

---

### 2. Tracer Bullet

まずシステムの一つの振る舞いを確認するテストを一つだけ書く。

```text
RED:
  最初の振る舞いのテストを書く
  ↓
  テスト失敗

GREEN:
  通すための最小コードを書く
  ↓
  テスト成功
```

これが Tracer Bullet である。

システム全体の経路が端から端まで機能することを確認する。

---

### 3. 増分ループ

残りの振る舞いについて繰り返す。

```text
RED:
  次のテストを書く
  ↓
  失敗

GREEN:
  通すための最小コードを書く
  ↓
  成功
```

ルール:

* 一度に一つのテストだけ
* 現在のテストを通す最小限のコードだけを書く
* 将来のテストを先回りしない
* テストは観測可能な振る舞いに集中する

---

### 4. リファクタリング

すべてのテストが成功した後で、
リファクタリング候補を検討する。

* [ ] 重複の排除
* [ ] Deep Module 化
* [ ] 自然な形で SOLID 原則を適用
* [ ] 新しいコードから既存コード改善のヒントを得る
* [ ] 各リファクタリング後にテスト実行

### 重要

**RED 状態でリファクタリングしてはいけない。**

まず GREEN に到達すること。

---

## 各サイクルのチェックリスト

```text
[ ] テストは実装ではなく振る舞いを記述している
[ ] テストは公開インターフェースのみを利用している
[ ] 内部リファクタリング後もテストが生き残る
[ ] コードは現在のテストを通す最小限である
[ ] 推測による機能追加をしていない
```
